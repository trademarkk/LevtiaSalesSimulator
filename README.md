# LEVITA Sales Simulator

Тренажер продаж для администраторов студии LEVITA:

- администратор проходит тренировку в формате живого диалога после пробного занятия;
- руководитель управляет библиотекой возражений, администраторами и базовым prompt тренажера;
- master управляет руководителями и инвайтами;
- сценарии и настройки хранятся в PostgreSQL / Supabase.

## Что внутри

- `Next.js 16` для интерфейса и API;
- `PostgreSQL` через `pg` (в текущей конфигурации — Supabase pooler);
- единый AI-движок в `lib/trainer-engine.ts` для сценариев, поточного диалога и итоговой оценки;
- `OpenAI`, `OpenRouter`, `DeepSeek` и `demo`-режим;
- cookie-based auth с HMAC-подписью сессии.

## Актуальная архитектура

### Канонический AI-движок

Основной и актуальный модуль тренажера — `lib/trainer-engine.ts`.

Он отвечает за:
- сбор сценария тренировки;
- потоковый ответ клиентки;
- итоговую оценку после завершения диалога;
- fallback-режимы;
- выбор AI-провайдера (`openai`, `openrouter`, `deepseek`, `demo`).

Основные маршруты, которые его используют:
- `app/api/scenario/route.ts`
- `app/api/chat-stream/route.ts`
- `app/api/chat/route.ts`
- `app/api/prompt-preview/route.ts`

### Legacy-файлы

Эти файлы сохранены как исторические/временные заглушки и не являются каноническим ядром:
- `lib/trainer.ts`
- `lib/trainer-service.ts`

Также в проекте есть несколько поколений UI-компонентов тренажера. Перед чисткой нужно отдельно зафиксировать, какой компонент считается боевым.

## Роли в системе

- `admin` — проходит тренировки и сохраняет тренировочные сессии;
- `manager` — управляет администраторами, возражениями и своим trainer prompt;
- `master` — управляет руководителями и инвайтами для регистрации руководителей.

## Быстрый старт

1. Установите зависимости:

```bash
npm install
```

2. Создайте локальные переменные окружения:
- `.env.local` — реальные секреты и локальные значения;
- `.env.example` — только шаблон без секретов.

3. Запустите проект:

```bash
npm run dev
```

4. Откройте [http://localhost:3000](http://localhost:3000).

## Переменные окружения

### Обязательные для реальной работы

- `DATABASE_URL` — строка подключения к PostgreSQL / Supabase.
- `SESSION_SECRET` — секрет для подписи сессии. В production обязателен.

### AI-провайдеры

- `AI_PROVIDER` — `openrouter`, `deepseek`, `openai` или пусто для fallback-логики.
- `OPENROUTER_API_KEY`
- `OPENROUTER_BASE_URL` — по умолчанию `https://openrouter.ai/api/v1`
- `OPENROUTER_MODEL`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL` — по умолчанию `https://api.deepseek.com`
- `DEEPSEEK_MODEL`
- `OPENAI_API_KEY`
- `OPENAI_MODEL`

## Важные замечания

- Файл `.env.local` должен содержать реальные секреты и не должен попадать в git.
- Некоторые служебные скрипты требуют, чтобы переменные читались не только из `.env`, но и из `.env.local`.
- Маршрут `POST /api/manager-invite` отключен. Для создания инвайтов руководителей используйте защищенный `POST /api/master/invites`.

## Что стоит сделать перед production

- настроить rate limit для login / register / invite flows;
- убедиться, что нет публичных административных маршрутов без role-check;
- привести README, scripts и env-слой к единому стандарту;
- зафиксировать канонические UI-компоненты тренажера и позже убрать legacy-слой;
- при необходимости декомпозировать `lib/db.ts` на тематические модули.
